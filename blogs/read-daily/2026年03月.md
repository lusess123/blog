# 20260320 What is Human In The Loop with AI? How HITL Shapes AI Systems
+ [https://www.youtube.com/watch?v=9iS-YYLIXiw](https://www.youtube.com/watch?v=9iS-YYLIXiw)
+ **human-in-the-loop** ：人在 AI 系统的工作流里，应该以某种方式被嵌进去。更重要的是，它不是一个非黑即白的开关，而是一条“人类参与程度”的光谱；在这条光谱的一端，就是最严格意义上的 **HITL**。
+ **human-on-the-loop**：也就是 AI 自主运行，但旁边仍有人持续监控，并保留否决权；一旦情况开始变糟，就可以按下“急停开关”，或者直接强行接管系统。
+ **human-out-of-the-loop**，也就是完全自治，整个过程根本没有人参与。AI 会自己感知、自己决策、自己行动；
    - 更适合出现在那些快得离谱的系统里，比如高频交易
+ **active learning**。它的基本思路是，先用一小部分已经标注好的数据训练模型，然后再让模型回头去看那些我们还没来得及处理的大量未标记数据。
+ 当这种规则无法被精确写出来时，办法就是让人直接“示范”给它看，这种方法就叫 **RLHF**，也就是 **reinforcement learning from human feedback**。
    - 在 **RLHF** 里，模型会针对同一个 prompt 生成两个回答，比如 A 和 B，然后由人来比较这两个选项，判断“这次 A 更好”。这里的人并不是在打标签，也不是逐字纠错，而是在表达一种偏好。
+ **confidence thresholds**：模型会给自己的每次预测打一个“确信度分数”，一旦低于预设阈值，就把这一项转交给人处理。于是 AI 负责相对简单的 90%，人来接住不确定的 10%。
+ **approval gates**：AI 先提出一个动作，必须有人明确点头，系统才能继续。
+ **escalation queues**：常规情况 AI 可以端到端处理，但遇到边缘案例就升级给人审查。
+ 代价：
    - 可扩展性。只要流程里多一个人工触点，就多一个“人形瓶颈”；如果系统每秒要处理成千上万次决策，人就必须非常策略性地决定自己该插在哪一步。
    - 是一致性：同一份数据，不同的人可能标出完全不同的结果，因为疲劳、偏见、主观性本来就是人的特征。
+ **human-in-the-loop** 的目标，从来不是把人永远拴在流程里，而是在系统还没建立起足够信任之前，让人继续守在那里。等系统逐渐可靠，它就会沿着我们前面说的那条光谱往前移动：从 **human-in-the-loop**，到 **human-on-the-loop**，再到某些场景下的 **human-out-of-the-loop**。这就是几乎每个 AI 部署都会经历的成熟曲线。

# 20260319 wtf is Harness Engineer & why is it important 下
+ [https://www.youtube.com/watch?v=kJPvfoLtFFY](https://www.youtube.com/watch?v=kJPvfoLtFFY)
+ 首先，他们会先准备一个 **initializer agent**，用专门设计的 prompt 让模型把初始环境搭起来
+ 然后，在之后的每一轮 session 里，会有一个 **coding agent** 让模型持续做小步推进，并留下结构化更新
+ 他们会先做一份 **feature list** 文档，避免 agent 一上来就把整个 app 一口气做完，或者过早认为项目已经完成；相反，会先把项目拆成 200 多个 feature，并固定在本地 **JSON** 文件里。
+ 默认情况下，所有任务一开始都标成失败，这样模型就会被迫始终回头看整体项目目标和当前进度，先挑最高优先级的任务，再继续做下一步。
+ 他们发现最有效的做法，是要求模型把自己的进展提交到 **Git**，写清楚描述性的 commit message，并在进度文件里再补一份总结。
+ 模型天然有一种倾向，就是在没有充分测试的情况下把事情标成“已完成”。
+ 给模型配上了能自己做 **end-to-end test** 的工具，比如 **Puppeteer MCP** 或 **Chrome DevTools**；这样 agent 就能发现并修掉那些从代码表面根本看不出来的 bug。
+ 把 **AGENTS.md** 当成“目录页”来用。架构设计、执行计划、数据库结构、产品规格、前端方案、安全说明等内容分别拆开存放，再把索引放进 **AGENTS.md**，这样 agent 在需要的时候就能按需取回相关信息，这其实就是 **progressive disclosure**。
+ 从 agent 的视角看，只要某个信息在环境里不可访问，那它实际上就等于不存在。
+ 这类架构，在传统软件公司里往往要等团队扩张到几百名工程师以后才会认真补上。但在 **coding agents** 的场景里，它反而成了很早就必须具备的前置条件；因为一旦边界先定清楚，团队和 agent 就能在不被细节微管理的前提下自由表达方案，也不用太担心架构逐渐漂移。
+ 与此同时，他们还大幅改进了代码库本身。比如让 app 可以按 **Git worktree** 独立启动，这样 **Codex** 就能同时拉起并驱动多个实例；又把 **Chrome DevTools Protocol** 接进 agent runtime，让 agent 能借助 **DOM snapshots**、截图和页面导航来复现 bug、验证修复。
+ 我们在做 agent，尤其是垂直领域 agent 时，往往会本能地去造很多专用工具；但真正的经验却是，大语言模型几乎总是更擅长使用它们天然更熟悉的通用工具。
+ 很多人都经历过，这类系统虽然“能跑”，却通常很脆弱、很慢，而且需要持续维护，因为每冒出一个新的边缘情况，你都得继续往 agent 里塞新的 prompt。
+ 根本原因在于，这些大模型对这类“代码原生工具”的熟悉程度，远高于那些需要它现生成的定制 **tool-calling JSON**。
+ 这种简洁架构能够成立的前提，依然是一套好的上下文与文档环境，让模型能用通用工具按需、逐步地取回上下文

+ 怎样设计一个能跨多个 session、多个 agent 协作运转的系统；怎样设计合适的工作流，让每个 session 都能把相关上下文重新取回来；又怎样选对工具，把模型的能力尽可能榨出来。

    - 对于长程任务 agent 来说，系统设计里最关键的是创造一个“可读、可理解”的环境，让每个子代理或新 session 都能迅速知道项目当前进展到哪了。
    - 验证极其重要。
    - 我们要更敢于信任模型本身。
+ 所谓“更信任模型”，意思不是一上来就把大量推理逻辑和业务规则封进专用工具里，而是应该先给模型尽可能多的上下文，再配上它原生就理解的通用工具，让它像人一样去探索、判断和行动
+ 第一个失败模式，就是 agent 特别容易“一口气做太多事情”。

+ 过去你们做的转录模型只负责转录，本身缺少 **LLM** 那种通用智能，所以大家通常是把转录模型和 **LLM** 组合起来用。而这个新模型`Universal 3 Pro`，像是把二者融合到了一起。它仍然是转录模型，但里面已经注入了 **LLM** 的通用理解能力。
