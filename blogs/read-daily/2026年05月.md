+ 当前沿实验室训练这些 **LLMs** 时，本质上是在构建巨大的强化学习环境。模型会得到“验证奖励”，也正因为这种训练方式，它们最终会不断进步，同时形成一种能力很不均匀的“锯齿状实体”。
+ 最先进的 **Opus 4.7** 一边能重构 10 万行代码库，或者发现零日漏洞，另一边却告诉我要走路去洗车店。这太离谱了。
+ 只要这些模型仍然保留这种锯齿化特征，它就说明两件事之一：
    - 第一，也许某些地方确实有点不对劲；
    - 第二，你需要稍微参与到流程里。你得把它们当作工具，也必须持续关注它们到底在做什么。
+ 所有关于可验证性的写作，其实都是在试图理解为什么这些东西会能力参差不齐。这里面有没有某种模式？我认为它大概是“可验证”加上“实验室在意”这两者的某种组合。
+ 我要强调这个维度：我们在某种程度上受制于实验室在做什么、它们碰巧把什么放进训练组合。
+ 如果你有大量多样化的数据集、强化学习环境等等，你就可以拉动这个杠杆。你可以使用自己喜欢的微调框架，启动这一套流程，然后得到一个实际效果相当不错的东西。
+ 最终几乎所有事情在某种程度上都能被做成可验证的，只是有些更容易，有些更难。因为即使是写作之类的任务，也可以想象用一组 **LLM** 评委来评判，并且大概能从这种方法中得到还算合理的结果。
+ vibe coding 关乎为所有人提高下限，agentic engineering 关乎保留专业软件过去已有的质量标准。一个是在提高下限，另一个是在向更高能力外推。
+ 招聘可能就该长这样：观察人在这种场景中的表现，看他们如何构建更大的项目，以及如何使用工具链，这大体上就是他会重点看的东西。

# 20260506  Andrej Karpathy: From Vibe Coding to Agentic Engineering 1

+ **LLM** 是一种新的计算机。它不只是更好的软件，而是一个全新的计算范式。**Software 1.0** 是明确写出来的规则，**Software 2.0** 是学习出来的权重，而 **Software 3.0** 就是现在这种东西
+ **Software 3.0** 大致指的是：你的编程方式现在转向了 **prompting**。而 **context window** 里放了什么，就成了你控制这个解释器的杠杆；
+ **Software 3.0** 的范式要原始、直接得多。你的神经网络正在承担越来越多的工作，而你的 prompt 或上下文可能只是一张图片，输出也是一张图片，中间根本不需要再有任何 app。个解释器就是 **LLM**，它会解释你的上下文，并在数字信息空间中执行计算。
+ 我一直试图回到这个问题：不只是问“哪些已有的事情现在可以做得更快”，而是要看到一些以前根本不可能实现的新机会。我甚至觉得，后者更加令人兴奋。
+ 有什么东西，未来回头看会显得完全显而易见，但今天却基本还没有被构建出来？
+ 很多事情会反过来：神经网络会变成类似“主进程”的东西，而 **CPU** 会变成类似“协处理器”的东西。
+ 神经网络完成大部分重活，而它们只是在某些确定性任务上使用工具调用，把工具当作一种历史遗留的附属物。真正掌控整个系统运行的，是这些神经网络。
+ 谈一谈 **verifiability** 这个概念，也就是这样一个事实：在那些输出结果可以被验证的领域里，AI 会更快、更容易地实现自动化。

# 20260505    Is this the only skill left?
[https://www.youtube.com/watch?v=7zCsfe57tpU](https://www.youtube.com/watch?v=7zCsfe57tpU)

+ 1985 年，丹麦计算机科学家 **Peter Naur** 写了一篇论文《Programming as Theory Building》这篇论文提出了一个简单但很有意思的观点：代码本身并不是程序
+ 真正的程序，是存在于程序员脑子里的东西：各个部分如何连接，为什么这样连接，以及如果拿掉其中一个部分会发生什么。那才是程序。按 **Peter Naur** 的说法，代码只是它的影子。40 年后的今天，我们有了 **AI coding agents**，可以按需替我们生成这个“影子”。
+ 发布自己不懂的 **AI** 生成代码时，我们迟早要付代价
+ 系统不只是把一堆零件放在一起。系统是一种模式：这些部分如何随着时间相互影响。你改变其中一个，其他部分就会反应；漏掉一个连接，整个模式就会断裂。这些概念来自 **systems dynamics**。
+ jagged frontier：这项技术在某些地方非常锋利，在另一些地方却意外迟钝，甚至同一次使用里都可能这样。知道这些边界在哪里，比如模型擅长什么、会悄悄搞错什么，现在已经是开发者核心技能的一部分，也是新的素养
+ systems thinking 的三个问题：
    - the true of the source : state 存在哪里？也就是系统里的真相由谁拥有？如果两个部分都以为自己拥有真相，那你已经有 bug 了，只是还没触发它而已。
    - feedback 存在哪里？是什么告诉你系统是否正常运行？你怎么知道？日志、指标、错误，都应该在某个地方冒出来。如果没有任何东西告诉你这些，那系统并不是在工作，很可能只是看起来像在工作。
    - 如果我删掉这个，会坏掉什么？在动手之前，你能不能在脑子里追踪系统中任何组件或部分的 blast radius(影响范围)
+ 为什么AI不是新的抽象层：
    - 编译器可不理解也信任；**LLM** 必须理解它做了什么，才能信任。它们不是同一种工具。只有下层可验证时，“抽象层”论点才成立
    - 每个高级开发者，包括我，都是在公开失败中练出系统思维的：先设计错系统，再承受后果。
    - **AI** 并没有拿走压力，它拿走的是那种挣扎。以前，那种痛苦本身就是课程。
+ 100 年前，体力劳动逼得大家更健壮；今天，环境不再替我们训练，普通人就没那么健壮。但今天的精英运动员，可能是史上体能最强的人类，因为“刻意训练”已成方法论，过去不是这样。编程也在走同样的路。2026 年普通初级开发者的系统理解，可能不如 2010 年。但刻意训练、选择自己做饭或举铁的人，会比我见过的任何初级开发者都更突出。
+ 如何训练系统思维？
    - 在提示之前先设计。
    - 把规格说明当作脚手架
    - 做“删除测试”

# 20260504  Recursion Is The Next Scaling Law In AI 上
[https://www.youtube.com/watch?v=DGtUUMNYLcc](https://www.youtube.com/watch?v=DGtUUMNYLcc)

+ **AI** 研究中一个很有意思的新趋势：递归。重点是如何在推理阶段用递归提升模型推理能力，而不是只把模型越做越大。2025 年有两篇论文让这种方法的威力变得很清楚：**HRM** 和 **TRM**。
+ **RNN** 是一种会被一次又一次递归调用的模型。
+ **gradient checkpointing**。但这本质上是在用更长的实际运行时间和更多计算，换更少的内存
+ **GPT-2** 是里程碑式架构和论文，核心就是不断预测下一个 token。
+ 在 **HRM** 和 **TRM** 论文里，作者把数独当作不可压缩问题。类似地，迷宫也是不可压缩问题，滚动求和也是不可压缩问题。
+ 想比 **n log n** 更快，通常需要访问某种外部记忆缓存。如果有一条可写入的“纸带”，就能选择性地把东西放到记忆里，从而更快。他怀疑这正是 **LLM** 的关键限制：模型内部没有内建的外部记忆纸带。

# 20260503 Cursor Worktrees: Deleting 12K LoC Code with 200 LoC Skills — David Gomes
[https://www.youtube.com/watch?v=WE_Gnowy3uw](https://www.youtube.com/watch?v=WE_Gnowy3uw)

+ Git里的 worktrees，可以把它们理解成独立的 checkout。它们就像你仓库的多个独立工作副本，让你可以并行工作
+ 这个功能最酷的一点是，你可以把同一个任务同时交给不同模型，然后比较它们面对同一个 prompt 会做出什么结果。我们把它叫作 best event，本质上就是让不同模型在同一任务上竞争
+ 可以用两个基础能力来让 **Cursor** 用户有效地使用 **worktrees**。一个是 agent skills，另一个是 sub-agents
+ 我们必须指示模型待在那个 **worktree** 里。我们真的要非常明确地告诉它：“不要在这个范围之外工作，也永远不要逃出去。”本质上，我们是通过比较强硬的 prompting 来做到这一点。
+ 这些 commands 在 **Cursor** 中的工作方式和 skills 极其相似：只有当用户选择加载它们时，相关 prompts 才会进入上下文。我们把它做成 commands 而不是 skills，唯一原因是这些 prompts 可以由我们的服务器和后端控制。
+ 他们可以先聊某件事，如果后来决定想在旁路环境里处理，就可以用 slash worktree 做到。
+ 新 skill 实现还有一个优点：在 **best event** 结束时，判断哪个模型做了什么的体验要好得多。parent 现在对每个 sub-agent 做了什么拥有更多上下文。用户甚至可以要求 agent 把不同实现里的不同片段拼接起来，而这以前是不可能的
+ 缺点：
    - 尤其是在很长的 session 里，模型很可能会忘记自己应该在哪个位置操作。有时候这些模型，特别是不太好的模型，会有点 hallucinate，或者变得有些失控，然后开始做不该做的事。
    - 它感觉更慢，因为你会看到 agent 创建 **worktree**，而且这些过程会出现在聊天里。它实际上并没有更慢，但确实会让人感觉 agent 在浪费时间做一些本该提前帮它做好的事情
+ 这是一个高级 power-user 功能，总体上我们可以接受它不那么容易被发现
+ 我们会用两种方式来改进。第一种是 evals，然后用这些 evals 改进 prompts。第二种是通过 **RL** 和训练。
+ **Braintrust** 这样的工具，写这类 eval 其实非常容易。你几乎不需要了解 evals，只要 prompt agent，它就会帮你完成所有事情。
+ **Cursor CLI**。它是 headless 的，所以非常适合 evals。然后我有两个 scorers：一个检查模型是否按预期在自己的 **worktree** 里做了工作；另一个则反过来检查模型是否在 primary checkout 里做了不该做的工作。
+ Cursor 3.0的一部分是一个更 agentic 的编程界面：你仍然可以编辑代码、查看代码，但 **UI** 和 **UX** 会更多围绕 agent 和聊天界面来优化
+ 我们正在研究一些新的本地并行化方案，它们不依赖 **Git**，也不依赖 **Git worktrees**。
