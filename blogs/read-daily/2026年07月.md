+ 发布上线正是真正工作开始的时刻。因为你必须尽快“闭环”：上线之后，要能对系统有控制感，也要理解它到底在发生什么。
+ 上线之后会发生什么？这并不新鲜，传统软件里也有同样的问题：要监控发生了什么，理解系统行为，保留日志来发现并修复问题。
+ 你给出指令，它们就能处理；但你不可能提前写出所有对话。于是问题进入更细的一层，也是让我睡不着的一层：你会失去对自己系统的手感。 
    - 新问题在于：**LLM** 不是确定性的，同样输入可能走出不同路径，输入稍微变一下，轨迹也会变。
    - 第二个、更可怕的问题是：失败会把自己藏起来。想象一个 **agent** 跑了很久，中途遇到问题，但运气好恢复了，找到了一些绕路方案，又试了别的工具调用。
    - **Anthropic** 也提到过，有时智能体会把功能标记为完成，却没真正检查是否可用。
+ 所以你会发现：运营一个 **agent**，本身也是一个 **agent** 问题
+ 大多数 **agent** 都很急着发 **PR**，很喜欢修问题。我们更倾向于用一个独立智能体，它有全新的上下文，从不同角度检查 **PR**，运行聚焦测试，而且不受原问题偏见影响，专门挑这个 **PR** 的毛病。
+ **PR agent** 和 **review agent** 每天发出的 **PR** 数量，是我们三个人的十倍。所以我们必须有一套清晰机制来处理这些 **PR**，避免自己成为系统瓶颈。以我的经验，这些智能体能提供很好的描述。

+ 组织往往不知道怎样构建好的 **skills**，也不知道如何把自己的操作流程转成 **agent** 能执行的东西。如果做不到这一点，就很难获得 **skills** 能带来的价值回报。
+ 一套共同的评分标准或框架来审视并改进它, skill checklist:
    -  **skill** 的 **trigger**，也就是它如何被调用，以及这里需要做哪些设计决策
    - **skill** 的内部结构，也就是它内部究竟如何组成、如何布局
    - 你如何真正通过 **skill** 来引导行为
    - 怎样让 **skill** 尽可能小？因为一旦我们有了一个可工作的 **skill**，接下来就要优化它，剪掉所有无关内容，剪掉所有 **no-ops**。
+ 一些有用的技巧：
    - 决定 **user invoked skills**，还是**model invoked skills** ？
        * 用户需要记在脑子里的东西越多，对“驾驶员”的能力要求就越高。
        * 看起来要么应该压低 **model invoked skills** 的数量，要么干脆全部用 **user invoked skills**。但用户调用的 **skills** 也有另一种负担：越多，用户的认知负担越高。
        * 这样你得到的结果是：你直接让一整类问题不再成为问题。因为这种不可预测性会迫使人们去评测自己的 **skills**，确认它们是否在正确的时间被调用；
    -  **skill** 的内部布局：
        * 两个主要单元：**steps** 和 **reference**
        * 从零开始写 **skill** ：先弄清楚是否需要步骤，然后写出这些步骤，再弄清楚这些步骤需要什么参考资料，并把参考资料放到 **skill** 里单独的位置。
    - 非常重要的约束:
        * 希望主 [skill.md](http://skill.md) 文件尽可能小
            + 一个很有用的方法：思考这个 **skill** 的不同分支，也就是它可能被使用的不同方式。把分支性的参考材料藏在 **context pointers** 后面
    - **steering** 
        * 解决的问题是：**agent** 不按我想要的方式做事。
        * **leading words**，或者如果你喜欢文学理论也可以叫 **lightvert**，它的想法是：有些词能在很小的空间里塞进大量意义。
        * 英语是一套非常宽的 **API**：你可以调用很多不同“函数”，尝试很多不同表达。外面有很多可用的 **leading word candidates**，而 **agents** 其实也很擅长帮你想这些词。
    - 通过隐藏未来目标和未来步骤，让当前步骤获得更多 **leg work**
        * 我在每一种试过的 **plan mode** 实现里都发现，“问澄清问题”这一步做的 **leg work** 总是不够。
        * 例子：**grill with docs** 现在是一个独立 **skill**，**agent** 只能看到流程中的这一部分。等 **grill with docs** 完成后，我们再去做 **two PRD**。换句话说，我们仍然有第一步和第二步，但 **agent** 一次只看到一步。
    - **pruning**：
        * **skills** 变大的另一个原因是 **sediment**。所谓沉积，就是多人一起维护同一套文档时很常见的现象：大家不断往一个共享 **Markdown** 文件里添加内容。
        * **no-ops**：它们看起来像是在做什么，但在这个 **skill** 的上下文里，并不会真正影响 **agent** 的行为。
        * 做 **deletion tests**，确保把内容压缩进 **leading words**，不要放无关内容，也不要留下沉积物
