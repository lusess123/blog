# 20260422 The Friction is Your Judgment — Armin Ronacher & Cristina Poncela Cubeiro, Earendil
+ [https://www.youtube.com/watch?v=_Zcw_sVF6hU](https://www.youtube.com/watch?v=_Zcw_sVF6hU)
+ **Flask**（一个 Python 框架），讽刺的是它被大量训练进了模型权重里，以至于现在很多人是因为 AI 在生成 Flask 代码才开始学它。
+ 我得坦率承认，我并不觉得我们已经有全部答案了。既获得了巨大的**杠杆**，也尝到过很大的失望，反复碰到的两个问题：
    - 心理学问题
        * 最关键的一点，我觉得我们没怎么意识到：因为我们快速产出大量代码，我们被骗了，以为自己真的变得更高效、做了更多工作。
        * 事实恰恰相反——因为我们现在反而没有那么多时间去真正停下来、思考、**设计**自己在做的事。当你处于这种**心流**里时，让自己停下来是非常困难的。
    - 工程挑战
        * 与写代码的主体——人和机器加在一起——的数量已经超过了那些能真正承担责任的人。结果就是越来越多的代码审查被跳过、被**走过场批准**。
        * 在工程层面，我们在做的是创建**更大的 PR**，制造这些巨大的变更，因为现在这几乎零成本。
        * AI制造的**失败情形**比人类写的代码要多得多。作为人类，你写这种代码时会隐隐觉得难受，而 agent 没有这种理由。如果你做出这种**勉强运行**、还乐于从局部故障里自行恢复的服务，你实际上就造出了极其**脆弱的系统**。
        * 这也意味着你很快就会搞出一个代码库——它的体量和复杂度让 agent 自己都已经**挖不出来了**。整套机器长时间运转，会在源代码里制造出比人类主导时多得多的**熵**。
    - 我们发现 agent 真正擅长的事情，和我们让它们去工作的**代码库类型**之间存在某种相关性。这里最典型的对比是**库 vs. 产品**。对于库来说，它们表现得好得多。这说得通——因为本质上，当你写库时，你要解决的问题定义非常清晰。
    - 就像过去你做任何系统设计一样，你的代码库现在已经变成了**基础设施**。因此你要把它设计成对 agent 也**易读**的样子，让它能充分利用。
    - 尽量**遵循所有已知模式**——因为对抗强化学习（RL）毫无意义。**不要有隐藏的魔法**
    - 案例规则
        * 不准用 裸 catch（毫不分类捕获全部异常的反模式）
        * 尽量让所有 **SQL** 都走**同一个查询接口**
        * 尽量为 UI 提供**一套原语组件库**
        * 不使用任何**动态导入**。我们强制**函数名唯一**
        * 「**机械性 bug**」——比如它明显违反了 **agents.md** 的规定。然后，我们会专门标出那些**需要人脑重新被激活**的变更。
        * 如果涉及**权限变更**，你最好自己思考，而不是交给 agent，因为这些事往往**文档不全**
    - 客观地评判代码库现状，在心理上极其困难。
    - 我工作过的每支工程团队都在说：「我们要消除交付中的摩擦」——这话没错。但如果你做过足够大规模的工程，就会知道 **SLO（服务水平目标）****正是一种被****有意设计来在工程流程里引入摩擦**的好系统，它逼你去想：我真的需要这种可靠性吗？这个服务真的这么关键吗？我的人手够运行它吗？
    - 自从有了 agent，我们渐渐形成一种观念：这些摩擦都该去掉。可实际上，我们很需要它。因为在物理层面上，**摩擦**恰恰是**转向**所必需的——**没有摩擦就没有方向控制**，这是不可或缺的。这正是**你的判断力**所在、**你的经验**所在——你要把它们插入进来，并开始**感受**它。

# 20260420 Hermes Agent
+ [https://hermes-agent.nousresearch.com/docs](https://hermes-agent.nousresearch.com/docs)
+ 一个越用越强的自主智能体
+ 关键特性：
    - closed learning loop（闭合学习循环）
        * Agent-curated memory 代理维护的记忆
        * periodic nudges 周期性提醒
        * autonomous skill creation 技能自主创建
        * skill self-improvement during use 使用过程中改进
        * FTS5 跨会话回忆（结合 LLM 摘要）
        * Honcho辩证用户建模
    - 环境在空闲时会进入休眠状态，几乎不消耗任何资源
    - 支持agentskills.io
    - 完全网页控制
+ Honcho
    - 解决的问题：代理会忘记事情、自相矛盾，或者在会话过程中丢失上下文
    - 问题不在于工程技术，而在于你没有从数据中提取出所有潜在信息。你需要进行穷举推理，处理矛盾，追踪长期的模式，并保持数据的一致性

# 20260418 PI
[https://github.com/badlogic/pi-mono](https://github.com/badlogic/pi-mono)

+ 你应该让 **pi** 来适配你的工作流，而不是反过来让自己迁就它
+ **Pi** 自带很强的默认能力，但它会刻意跳过一些功能，比如子代理和计划模式。
+ 会话以树状结构存储，所有分支保存在一个文件中。按消息类型筛选，将条目标记为书签。
+ **AGENTS.md**：项目指令文件，启动时从 `~/.pi/agent/`、父级目录和当前目录加载
+ 当接近上下文限制时，自动摘要较早的消息。通过扩展完全可定制：实现基于话题的压缩、代码感知的摘要，或使用不同的摘要模型。
+ skills：带有指令和工具的功能包，按需加载。渐进式展示，不会撑爆提示词缓存
+ 动态上下文：扩展可以在每轮对话前注入消息
+ Steer 和 follow up 两种模式的中间发送消息
+ 扩展：提供的是原语，而非固定功能。
